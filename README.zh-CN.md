## 📌 项目介绍

**big-event** 是一个基于 **Vue2 + Vue Router + Vuex + Element UI + Axios** 搭建的后台管理系统项目，主要围绕 **用户认证、文章分类管理、文章发布管理、个人中心维护、首页数据可视化展示** 等业务场景展开。

项目整体以“内容管理后台”为核心定位，强调 **功能完整性、交互流畅性、模块拆分合理性以及基础工程规范**。在项目实战过程中，我重点完成了登录注册、文章管理、用户中心以及部分页面交互与样式优化工作。

## 🚀 1. 项目成果介绍

### 1.1 最终成果展示

项目最终形成了一套可正常运行的后台管理系统，主要页面包括：

- **登录页 / 注册页**：支持用户注册、登录与登录态接续。
- **首页**：通过 ECharts 展示文章与评论相关数据图表。
- **文章分类页**：支持分类的新增、编辑、删除。
- **文章列表页**：支持筛选、分页、删除、跳转编辑。
- **文章发布页**：支持富文本编辑、封面上传、发布文章、存为草稿。
- **个人中心页**：支持基本资料修改、头像上传、密码重置。

### 1.2 功能简述

本项目主要实现了以下核心功能：

- **用户认证**：完成注册、登录、token 存储与路由访问拦截。
- **权限控制**：未登录用户无法进入业务页面，登录失效时统一跳回登录页。
- **文章分类管理**：对文章分类进行增删改查。
- **文章管理**：支持文章列表展示、分类筛选、状态筛选、分页查询、删除与发布。
- **富文本编辑**：基于 `vue-quill-editor` 实现文章内容编辑。
- **用户信息维护**：支持昵称、邮箱、头像、密码等信息更新。
- **数据可视化**：首页借助 `ECharts` 展示统计图表，增强后台展示效果。

## 🧩 2. 模块功能介绍

### 模块一：**文章管理模块**

文章管理模块是本项目中业务逻辑相对最完整、交互也比较集中的部分，主要由 **文章列表页**、**文章分类页**、**文章发布页** 三部分组成。

**核心亮点：**

- **筛选与分页联动**：文章列表支持按照分类、状态进行筛选，并结合分页参数统一请求接口，提升列表管理效率。
- **弹层复用表单组件**：文章发布能力被拆分为独立的 `article-form` 组件，既可以用于弹窗发表文章，也可以用于独立页面发布文章，提升了组件复用性。
- **富文本与封面上传结合**：在文章发布时，支持富文本内容输入、封面图片选择预览、`FormData` 提交，满足真实业务中的文章发布需求。
- **草稿 / 发布双状态处理**：通过统一提交方法传入不同状态，实现“**发布**”与“**存为草稿**”两种业务分支，减少重复代码。

**技术实现要点：**

- 使用 `Promise.all` 并行获取分类列表与文章列表，减少页面初始化等待时间。
- 对接口返回结果做了兼容处理，增强了前端对不同数据结构的适配能力。
- 对富文本内容进行了“去标签后判空”校验，避免只输入空标签也能提交的问题。
- 对封面图使用 `URL.createObjectURL` 生成本地预览，并在组件销毁时及时释放资源。

### 模块二：**用户中心模块**

用户中心模块页面集中，包含了 **基本资料修改、头像上传、密码重置** 三类不同表单能力。

**核心亮点：**

- **通过路由路径映射不同子功能**，同一页面组件根据路由状态切换不同表单区域，减少页面重复开发。
- **表单校验更贴近业务规则**，例如新密码不能与原密码一致、确认密码必须一致、邮箱格式必须正确等。
- **头像上传支持即时预览**，提升了用户交互体验。
- **用户信息与头像展示联动 Vuex**，保证顶部导航、侧边栏、个人中心页面中的用户信息展示保持一致。

**技术实现要点：**

- 使用 `computed` 结合路由路径映射当前激活 tab，避免多页面重复维护。
- 借助 `watch` 监听用户信息变化，实时同步头像展示。
- 通过 Vuex 持久化 token，保证刷新页面后登录态仍可恢复。

## 🛠 3. 问题总结

### 3.1 项目中遇到的问题

在项目开发过程中，主要遇到了以下几类问题：

- **接口返回结构不完全统一**，不同接口返回的数据层级存在差异。
- **登录态管理容易出现脏数据问题**，例如 token 为 `undefined`、`null` 或携带多余前缀。
- 文章发布与文章编辑场景存在重复代码风险。
- 分类管理、文章管理、个人中心中都存在表单弹层或状态切换逻辑，容易让代码变得分散。

### 3.2 问题的分析与解决

针对上述问题，我主要从“**先定位原因，再做修正**”的方式处理：

- 对接口数据返回不一致的问题，我没有直接在模板中硬编码取值，而是封装了数据提取方法，并在响应拦截器中统一做**数据剥离**。
- 对 token 问题，我在本地存储工具中增加了统一格式化处理，过滤无效值，并去除可能重复带上的 `Bearer` 前缀。
- 对重复代码问题，我将文章发布表单抽成独立组件，让不同入口都复用同一套表单逻辑。
- 对登录失效问题，我在 Axios 响应拦截器中统一处理 `401` 状态，自动清除 token 并跳转登录页。

### 3.3 更好的解决办法或其他方案

目前的实现已经能满足项目要求，但如果继续优化，还可以考虑以下方案：

- **将表单逻辑进一步抽象为组合型配置**，例如分类弹窗、用户资料表单都可继续提炼复用。
- **引入更严格的状态管理边界**，例如把用户登录、退出、资料更新后的状态同步都统一走 Vuex action。
- **增加请求 loading、异常兜底和空状态展示规范**，让页面体验更完整。

### 3.4 前端代码优化

在项目开发中，我做了以下前端代码优化：

- **组件复用优化**：将文章表单抽离成通用组件，减少重复开发。
- **请求处理优化**：通过请求拦截器统一带 token，通过响应拦截器统一处理错误与数据剥离。
- **状态持久化优化**：登录 token 存入本地并做合法性清洗，减少异常状态。
- **性能细节优化**：分类列表与文章列表初始化并发请求，缩短等待时间。
- **资源释放优化**：封面图预览在组件销毁前主动释放 URL，避免不必要的内存占用。
- **可维护性优化**：将默认表单、路径映射、数据提取等逻辑单独封装，降低模板与业务逻辑耦合度。

## 🔄 4. 需求问题应对处理

### 4.1 如何看待需求变更

我认为项目中的需求变更是很正常的事情，尤其在真实业务中，需求往往会随着使用反馈、产品理解加深、接口调整优化而不断调整。前端开发不能只停留在“把页面写出来”，更重要的是要让代码具备一定的 **可扩展性、可维护性和容错能力**，这样面对改需求时才不会被动。

### 4.2 从需求变更中得到的经验

通过这次项目实战，我的体会是：

- **前期命名和结构设计要尽量清晰**，否则后期一改需求就容易牵一发动全身。
- **不要把接口字段、页面逻辑、展示逻辑强耦合在一起**，否则改一点点需求就要大改。
- **尽量把公共能力抽离出来**，例如公共表单、公共请求封装、统一状态处理。
- 需求变化不可怕，可怕的是代码没有预留扩展空间。

### 4.3 代码如何复用才能提升效率

在本项目中，我对“代码复用提升效率”有比较直观的体会：

- **文章发布表单组件复用**：我把文章标题、分类、富文本内容、封面上传、提交逻辑封装在 `article-form` 中，这样在弹窗发布和独立页面发布时都可以直接复用，避免重复维护两套表单。
- **用户中心多功能页面复用**：个人资料、头像、密码三种功能共用同一个页面组件，通过路由切换显示不同区域，减少了多个页面的重复创建成本。
- **工具函数复用**：例如 token 的存取、接口请求拦截、数据提取逻辑，都属于可重复利用的基础能力。

我的体会是：**复用不是单纯把代码写少，而是让后续修改时只需要改一个地方。** 这对项目修改效率提升非常明显。

## ⚙️ 7. 命令说明

### 安装依赖

```bash
npm install --legacy-peer-deps
```

**`--legacy-peer-deps`** 的作用是忽略部分依赖包之间的同版本冲突限制，保证项目在当前依赖环境下可以顺利安装。

### 启动项目

```bash
npm run serve
```

### 打包项目

```bash
npm run build
```
